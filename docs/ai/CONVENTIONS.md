# Технические и стилевые правила

## Обязательный рабочий протокол

- Перед задачей прочитать все файлы в `docs/ai/`.
- Любое изменение кода сопровождать обновлением `CURRENT_STATE.md`.
- Архитектурные решения и их причины фиксировать в `DECISIONS.md`.
- После завершения задачи добавлять краткую запись в `CHANGELOG_AI.md`.
- Для новой функциональности добавлять автотесты; тесты должны выполняться при каждой сборке.
- При падении теста установить причину, исправить дефект и повторить сборку.
- Не хранить в памяти логи, stack trace, временные рассуждения и промежуточные гипотезы.
- Не ломать рабочий функционал и не проводить массовые рефакторинги без явной необходимости.

## Подтверждённые правила кодовой базы

- TypeScript работает с `strict: true`; исходники React используют `.tsx`.
- Компоненты оформляются именованными функциями и именованными экспортами.
- Компоненты секций находятся в `src/components/`, данные — в `src/data/`.
- Общие стили находятся в `src/styles.css`; именование CSS-классов семантическое, в kebab-case.
- Интерактивные и навигационные элементы снабжаются доступными атрибутами (`aria-label`, `aria-labelledby`, `aria-live`, `aria-selected`) там, где это требуется текущими компонентами.
- Контент интерфейса — на русском языке.
- Переменные frontend-окружения используют префикс `VITE_`; существующий код метрик и URL также поддерживает отдельные `NEXT_PUBLIC_*`.
- Секреты Telegram используют только server-only переменные без префикса `VITE_`.
- Изображения, доступные по корневым URL, хранятся в `public/images/`.
- Frontend-тесты находятся в `src/__tests__/`, серверные тесты — рядом со скриптами; оба набора используют Vitest.
